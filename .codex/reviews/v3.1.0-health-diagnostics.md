# Codex review - v3.1.0 health diagnostics (uncommitted)

Verdict: REQUEST_CHANGES

## MAJORs
1. Health entities do not reliably update on REPEATED failed refreshes:
   HA's DataUpdateCoordinator notifies listeners only on the healthy->failed
   TRANSITION, then suppresses notifications while already failed. Reading
   last_update_success from always-available entities misses repeats.
2. Live anomaly (logged UpdateFailed but entities frozen at prior success)
   most likely = a stale/different coordinator instance. Jittered scheduled
   refresh sleeps through asyncio.sleep; cancel_scheduled_refreshes cancels
   future callbacks, not an in-flight coro. Fix: pass config_entry=entry,
   await async_shutdown() on unload, track/cancel jitter tasks, add entry id
   to refresh logs.
3. First-run with no cache + failed bootstrap creates NO health entities and
   NO refresh button (entity setup gated on coordinator.data is not None).
4. "YW latest data date" (latest_data_date) runs ~1 day ahead of the newest
   real daily point - misleading. Derive a user-facing date from
   _latest_point().point_date instead, use SensorDeviceClass.DATE.

## MINORs
- ConfigEntryNotReady is wrong for post-setup bridge-unreachable; use UpdateFailed.
- Date-only diagnostics should be DATE, not midnight-UTC TIMESTAMP.

## Redesign guidance
- "Last update" = last attempt timestamp: good, no state_class.
- Do NOT use free-text Ok/error as main state. Use ENUM device_class with
  fixed options (ok, login_failed, bridge_unreachable, api_error,
  unknown_error, no_attempt); raw short error in an attribute.
- After cache restore there has been no poll: status = no_attempt/unknown,
  NOT ok, until the first real refresh.
- Own health state on the coordinator (last_attempt, last_success, status,
  last_error) in a wrapper around the fetch; notify health listeners on every
  completed attempt, including repeats.

## Tests
Current health tests bypass the real path (manually set last_update_success +
call async_update_listeners). Need real async_refresh() failure tests, repeated
failure, no-cache bootstrap failure, and latest-daily-reading-date-vs-marker.

## Follow-up review (after rework)
Verdict: APPROVE_WITH_NITS. All 4 MAJORs + both MINORs RESOLVED.
- Remaining MINOR (fixed): manual notify on first failure could emit a stale
  normal-entity event; now only notify manually when already failing
  (HA notifies the first transition itself).
- Residual NIT (out of scope, self-heals on reload): no-cache failed bootstrap
  does not dynamically add per-property entities until reload; health entities
  do appear.
